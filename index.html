ChatGPT vs DeepSeek

AI Showdown in the Battle of Mathematical Minds

A comparative feasibility study evaluating ChatGPT and DeepSeek on complex mathematical problems — from algebraic reasoning to multivariable calculus. Assessed on correctness, efficiency, consistency, reliability, and cost of maintenance.

Spring 2025 — AUB ENGL 206 — Dr. May Mikati 21-Page Report
Download Full Report (PDF) →
AI Showdown
ChatGPT vs DeepSeek
in Complex Mathematics
01 Research Overview

Motivation

AI tools have transformed mathematical problem-solving, providing solutions in seconds that historically required extensive collaboration. With multiple competing LLMs on the market, choosing the right tool for a given task is non-trivial. This study directly compares ChatGPT and DeepSeek on real mathematical problems spanning algebra and multivariable calculus.

ChatGPT

Black-box reasoning model excelling at concise, formula-based solutions. Faster responses, strong at explaining relevance of each step. Paid subscription for advanced models.

DeepSeek

Trained on 120B+ math-focused examples with GRPO optimization. Step-by-step educational style, geometric approaches, free access. 51.7% on MATH benchmark vs ChatGPT's 42-45%.

02 Methodology

Test Problems

Both models were tested on identical problems: a Grade 9 algebraic reasoning problem (system of equations with logical deduction) and a Calculus II surface area problem (paraboloid cut by a plane). Performance was evaluated through primary testing and secondary academic benchmarks.

03 Key Findings

Comparative Results

While both models achieved 100% accuracy, their approaches and trade-offs differed significantly. ChatGPT favored concise, formula-based solutions ideal for professionals, while DeepSeek provided educational, step-by-step breakdowns better suited for students.

04 Recommendations

Use ChatGPT when...

  • You need quick, concise answers
  • Time-sensitive contexts (exams, engineering)
  • You have a strong math foundation already
  • Multi-platform accessibility matters

Use DeepSeek when...

  • You need step-by-step explanations
  • Learning and tutorial environments
  • Budget constraints (free access)
  • Logic verification is more important than speed

Veridict

AI contract review platform with fine-tuned LLM and human-in-the-loop validation

Upload a contract and a staged pipeline parses it, retrieves policy evidence via RAG, runs two parallel AI lanes — Harvey (cross-policy conflicts) and Kira (intra-contract risks) — merges findings through Admin, and queues the result for a human reviewer to approve, redline, and export. Built with FastAPI, React/Vite, PostgreSQL/pgvector, and a QLoRA fine-tuned Gemma 4 model. Deployed on GKE with RunPod GPU inference, full CI/CD, and Prometheus monitoring.

May 2026 · AUB Karim Assi, Karl Gerges, Abdel Rahman El Kouche React · FastAPI · GKE · RAG · Fine-tuned LLM
View Live App →
01 Pipeline

Run-based pipeline

Each contract triggers a durable run through nine stages: create_run → ingest_pdf → parse_ocr_normalize → clause_index → harvey_context_load → harvey_review_block ‖ kira_review_block → admin_merge → awaiting_human_review → finalized. Harvey and Kira run at the same stage order — parallel with two workers, sequential with one. Every stage has its own status (pending, running, done, failed, retrying) so failures are isolated and retryable with a full audit trail.

Workspace dashboard

Contracts are workspace-scoped. The dashboard shows status counts (awaiting review, in-progress, finalized), risk level, version, and workspace tabs. 25 contracts in this view across Demo and General workspaces, each linked to its full review run.

02 AI Agents

Harvey & Kira — two lanes, one output

Role separation is the core engineering idea. Harvey owns cross-policy reasoning with RAG citations; Kira owns intra-contract risk detection. Neither trespasses on the other’s scope. Admin merges both into per-clause verdicts for the human reviewer.

Harvey — sequential 3-stage RAG lane

  • Stage 1 — Contradiction Finder: exhaustive pass against the policy knowledge base; flags every plausible conflict with a required RAG citation.
  • Stage 2 — Regression Challenger: receives stage-1 output and filters down to genuine material regressions — discards documented updates and equivalent restatements.
  • Stage 3 — Downstream Risk: enriches validated findings with enforcement-gap and liability analysis; cannot add or remove findings, only raise severity.

Kira — worker + panel improvement loop

  • KiraWorker analyzes contract text for ambiguity, missing protections, exploitable gaps, and imbalanced terms — each finding includes a concrete recommended rewrite.
  • 3 KiraPanelReviewers (baseline, never fine-tuned) vote approve/reject. 2-of-3 rejects triggers aggregated feedback back to the worker for revision.
  • Max 3 revision iterations, then KiraValidatorAgent signs off before passing to Admin. Baseline panel guards against fine-tuned worker drift.

Review output

A completed run on HC_RetainerAgreement.pdf produced seven findings — five high-risk, one warning, one low. The review page shows findings by severity, sign-off panel, export actions, original PDF download, and legal notes.

Document-grounded viewer

The contract viewer links each AI finding back to the exact clause. The right panel lists recommendations by severity; the left displays the contract page in context. Findings are not a free-form summary — they’re anchored to the document.

Finding detail

Expanded finding cards show the risk explanation, the exact clause quoted as evidence, a concrete suggested rewrite, and reviewer action controls. This example flags third-party sharing of client information and recommends requiring prior written consent.

03 Model Training

Knowledge distillation → QLoRA on Gemma 4

CUAD and MAUD provided raw legal source material. DeepSeek v4 Pro acted as teacher to enrich each example with risk type, severity, worst-case impact, explanation, and evidence spans. That labeled JSONL was exported into Gemma chat-format turns and used to QLoRA fine-tune Kira on unsloth/gemma-4-26B-A4B-it — training only adapter weights with the base model in 4-bit mode.

04 Deployment & MLOps

CI/CD pipeline

GitHub Actions runs backend tests, frontend build, Docker image build and push to Google Artifact Registry, an eval gate that blocks deploy if Kira metrics fall below threshold, GKE deployment, smoke test, and rollback — all passing.

Infrastructure

  • GKE — API, worker, frontend, PostgreSQL, ingress, migration job.
  • RunPod A100 SXM — Kira and base model inference via OpenAI-compatible endpoints (32 vCPUs, 250 GB memory).
  • Google Artifact Registry — Docker images for API, worker, and frontend.
  • Docker Compose — full local stack with optional local inference.

Monitoring

  • Prometheus — scrapes API (port 8000), worker (port 9100), Kira vLLM, and base model vLLM; 15-day retention, all targets healthy.
  • Eval gate — blocks deploys if model metrics regress below threshold.
  • slowapi rate limiting; JWT auth; workspace-scoped access control throughout.
